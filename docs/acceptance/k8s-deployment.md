# Core Feature Area A: Kubernetes Resource Deployment

Part of the [GA Acceptance docs](README.md).

## Goal

Users can deploy their own Kubernetes resources inside a Project by submitting a
ConfigFile.

A ConfigFile is a versioned Kubernetes YAML artifact. Every deployment
references an immutable ConfigFile version.

Users do not receive direct Kubernetes credentials. All apply/delete/watch
operations go through NexusPaaS APIs and the `k8s-control-service`.

## Supported GA v1 Resource Types

| Resource | GA v1 Policy |
|---|---|
| `Pod` | Allowed with security mutation and validation |
| `Job` | Allowed for batch workloads |
| `Deployment` | Allowed for long-running and WebRTC GUI workloads |
| `Service` | Allowed only inside Project namespace |
| `ConfigMap` | Allowed |
| `Secret` | Discouraged; allow only through platform secret API or ExternalSecret profile |
| `PersistentVolumeClaim` | Prefer platform-generated PVCs; user YAML may only reference approved PVCs |
| `ResourceClaimTemplate` | Platform-generated by default; advanced use requires admin capability |
| `Ingress` / `HTTPRoute` | Allowed only through NexusPaaS Gateway profile |
| `Namespace` | Forbidden |
| `ClusterRole` | Forbidden |
| `ClusterRoleBinding` | Forbidden |
| `RoleBinding` | Forbidden by default |
| `CRD` | Forbidden |
| `MutatingWebhookConfiguration` | Forbidden |
| `ValidatingWebhookConfiguration` | Forbidden |
| `StorageClass` | Forbidden |
| `PriorityClass` | Forbidden |
| `DaemonSet` | Forbidden by default |
| `StatefulSet` | Deferred; not GA v1 unless explicitly accepted |
| `CronJob` | Deferred; not GA v1 unless explicitly accepted |
| `Node` / `PersistentVolume` | Forbidden |

## ConfigFile Deployment Flow

```text
User / CLI / Web
  -> platform-gateway
  -> compute-api / workload-service
  -> ConfigFile parse and immutable version commit
  -> manifest preflight validation
  -> image-registry-service image allow-list check
  -> storage-service mount-plan check
  -> scheduler-quota-service Plan / Queue / Quota / time-window admission
  -> k8s-control-service server-side dry-run
  -> k8s-control-service apply
  -> Kubernetes admission: Kyverno / ValidatingAdmissionPolicy
  -> usage-observability attribution
  -> audit-compliance event
```

## Manifest Validation Requirements

| Category | Required Behavior |
|---|---|
| YAML parsing | Multi-document YAML must parse safely. Invalid YAML is rejected. |
| Resource allow list | Only GA-supported namespaced resources are allowed. |
| Namespace isolation | User-supplied namespace is rejected or overwritten with Project namespace. |
| Label injection | Platform injects `nexuspaas.io/user-id`, `nexuspaas.io/project-id`, `nexuspaas.io/group-id`, `nexuspaas.io/job-id`, `nexuspaas.io/queue`, and `nexuspaas.io/plan-id`. |
| Owner reference | Every created object must be traceable to a Job / ConfigFile version. |
| Image allow list | Every image must resolve to an allowed digest in the Project image allow list. |
| Resource limits | CPU and memory requests/limits are required. |
| GPU requests | GPU fields must match Plan, Queue, DRA, and MPS policy. |
| Security context | Default policy is restricted. Exceptions require Project capability. |
| Network | Default deny ingress/egress; egress exceptions require Project capability. |
| Storage | Only platform-approved PVCs, user storage, group storage, or Project storage may be mounted. |
| HostPath | Forbidden unless explicitly approved by platform admin. |
| Dry run | All manifests must pass server-side dry-run before apply. |
| Cluster admission | Kubernetes admission policy must reject bypass attempts even if platform API is bypassed. |

## Acceptance Criteria

| ID | Acceptance Criteria |
|---|---|
| K8S-001 | A user can submit a safe Deployment + Service YAML to an authorized Project and get a running workload. |
| K8S-002 | A user cannot deploy to another Project namespace. |
| K8S-003 | A ConfigFile version is immutable after deployment. |
| K8S-004 | A Job record always references the exact ConfigFile version used. |
| K8S-005 | Forbidden resources such as Namespace, CRD, ClusterRoleBinding, webhook configuration, StorageClass, and PriorityClass are rejected. |
| K8S-006 | ConfigFile validation returns clear machine-readable rejection reasons. |
| K8S-007 | All user workload resources are created only by k8s-control-service. |
| K8S-008 | Direct kubectl apply bypass attempts are rejected by Kubernetes admission policy. |
| K8S-009 | All deployed objects have NexusPaaS ownership labels and annotations. |
| K8S-010 | All created Kubernetes resources are recorded in Job status for cleanup and reconciliation. |
| K8S-011 | Failed apply operations are idempotent and can be retried without orphaning resources. |
| K8S-012 | Delete/cancel removes owned resources and releases quota. |
| K8S-013 | Reconciler can detect DB says running but Kubernetes object is missing. |
| K8S-014 | Reconciler can detect Kubernetes object exists but DB ownership record is missing and marks it for quarantine or cleanup. |
| K8S-015 | All ConfigFile submit/apply/delete operations generate audit events. |
| K8S-016 | All ConfigFile APIs have RBAC and Project membership tests. |
| K8S-017 | Admission tests include valid YAML, invalid YAML, forbidden resources, namespace spoofing, unsafe security context, hostPath, external egress, and image allow-list failure. |
| K8S-018 | Production cluster has Kyverno or ValidatingAdmissionPolicy installed and enforcing the same critical rules as platform preflight. |
