# P2 AcceleratorProfile DRA/MPS

## 1. Objective

Create the next small AI/HPC reviewable slice: `AcceleratorProfile` CRUD, seeded defaults, scheduler admission resolution, and workload dispatch injection of accelerator scheduling selectors/labels, while reusing the existing DRA/MPS path in `dispatcher_dra.go`.

## 2. Background

`docs/acceptance/gpu-dra-mps.md` marks GPU policy as the next highest active AC slice (`GPU-001` .. `GPU-018`), with live GPU evidence (`GPU-016`) deferred until hardware validation.

Prior AI/HPC slices are already in place:

- Storage data path (`P1`) landed.
- Network profiles landed and dispatch injects networking metadata.
- Placement profiles landed and dispatch injects scheduler/queue metadata.
- Existing code already validates fractional GPU usage and allowed GPU models, and already builds DRA `ResourceClaimTemplate` + MPS config.

This slice fills the missing policy gap: selecting accelerator policy through scheduler-owned profile metadata instead of ad-hoc per-workload hints.

## 4. Assumptions

- This plan targets implementation actions for the next reviewable slice; no code changes are included in this artifact.
- `scheduler-quota-service` owns accelerator policy and seeds defaults as records in `platform_records`.
- `workload-service` receives admission-reviewed metadata and only performs deterministic manifest injection; it does not own profile records.
- Existing generic record stack (`platform_records`) remains for this slice; no new SQL DDL or custom table.

## 5. Non-Goals

## 6. Current Behavior

- Admission validates quotas, plan limits, fractional GPU math, and allowed GPU models.
- Workload submit stores admission outputs that are already generated by scheduler-quota.
- DRA/MPS dispatch path exists when jobs request GPU markers and counts.
- There is no accelerator profile model for node-selector/label strategy, and no profile-based selector injection.
- Network/placement resolvers exist and are injected through dedicated dispatcher modules.

## 7. Target Behavior

### 7.1 Slice Goal

Admins can create and manage accelerator placement policy entities that resolve to scheduler-consumed scheduling metadata.

### 7.2 Required Acceptance Criteria

- `AcceleratorProfile` entities can be CRUD’d via admin API.
- Startup seeds conservative accelerator defaults exactly once.
- Workload submit accepts optional `accelerator_profile`.
- Admission resolves the requested profile and rejects:
  - unknown profile references,
  - disabled profiles.
- Admission output includes scheduler-resolved accelerator selector/label metadata.
- Workload dispatch injects resolved selector/label metadata into all supported workload templates (Pod/Job/Deployment/VCJob) via explicit dispatch ordering.
- No-op behavior when no profile is requested.

### 7.3 Data Shapes

- `accelerator_node_selector` (map string->string)
- `accelerator_labels` (map string->string)
- Optional MPS-related hint fields that map safely to current DRA admission constraints.
- Dispatch application order in scope:
  - Placement profile resolution must run before accelerator metadata merge.
  - Accelerator injection runs after placement and before streaming, networking, and DRA mutation.
- Merge/conflict behavior:
  - node selector merges are additive; conflicts fail fast during manifest preparation with an explicit conflict error.
  - accelerator labels are merged only for accelerator-owned keys and must not overwrite pre-existing non-accelerator labels.

## 10. API / Contract Changes

PR1 adds admin CRUD routes:

- `GET /api/v1/accelerator-profiles`
- `POST /api/v1/accelerator-profiles`
- `GET /api/v1/accelerator-profiles/{id}`
- `PUT /api/v1/accelerator-profiles/{id}`
- `DELETE /api/v1/accelerator-profiles/{id}`

PR1 API/contract fields for `scheduler.create_accelerator_profile`:

- `required_request_fields`:
  - `name`
- `optional_request_fields`:
  - `id`
  - `enabled` (default `true`)
  - `node_selector`
  - `labels`
  - `default_mps_sm_percentage`
  - `default_pinned_memory_limit`
  - `allowed_device_class_name`
- Request validation:
  - `name`: non-empty string, uniqueness per resource.
  - `enabled`: boolean when provided.
  - `node_selector` and `labels`: object map string->string; empty maps are valid when only MPS/device-class defaults are used.
  - `default_mps_sm_percentage`: integer `1..100` when present.
  - `default_pinned_memory_limit`: Kubernetes resource quantity string (for example, `8Gi`) when present.
  - `allowed_device_class_name`: non-empty string when present.
- Example POST request:

```json
{
  "id": "nvidia-gpu-default",
  "name": "NVIDIA GPU default",
  "enabled": true,
  "allowed_device_class_name": "nvidia.com/gpu",
  "node_selector": {
    "accelerator.nexuspaas.io/profile": "nvidia-gpu-default"
  },
  "labels": {
    "accelerator.nexuspaas.io/class": "nvidia"
  },
  "default_mps_sm_percentage": 50,
  "default_pinned_memory_limit": "8Gi"
}
```

## 14. Security Considerations

- Profile CRUD endpoints remain admin-gated.
- Workload authors can request a profile only; scheduler writes the resolved effective selector/labels.
- Reject profiles with invalid non-string selector/label maps; allow empty selector/labels for profile-only normalization (device class / MPS policy hints).
- Normalize maps as string->string only to prevent arbitrary injection.
- Keep DRA path unchanged for resource accounting/security decisions.

## 15. Implementation Steps

1. PR1: CRUD + seeded defaults in scheduler-quota
   - Register `accelerator_profiles` table/event/routes in `spec.go` + admin routes in `handler.go`.
   - Add required fields and defaults in a dedicated accelerator profile module.
   - Seed profiles idempotently with conservative defaults aligned to existing labels:
     - `nvidia-gpu-default`
     - `nvidia-h100-sxm-rdma`
   - Add CRUD/seed unit tests and contract fixture registration checks.

2. PR2: admission resolver + persistence
   - Extend admission decode and request structs with `accelerator_profile`.
   - Add resolver module to load and validate profile by id/name.
   - Populate `admissionReviewData` with resolved `accelerator_*` fields.
   - Extend job submit flow to persist resolved accelerator fields.
   - Add tests for success, missing-profile failure, disabled-profile failure, and no-op path.

3. PR3: workload dispatch injection
   - Add dedicated accelerator injector in `dispatcher_accelerator.go`.
   - Inject selectors and labels into all supported workload manifests: Pod, Job, Deployment, and VCJob.
   - Order within `prepareDispatchManifests`:
     1) placement/network resolve,
     2) accelerator injector,
     3) network/streaming/DRA injectors.
   - Merge logic:
     - node selectors append keys unless already present with a different value; any conflict is a hard manifest-preparation failure.
     - label merge applies only accelerator-owned labels (`accelerator.*` or `nexuspaas.io/accelerator-*`) and never overrides existing non-accelerator labels.
   - On selector conflict, fail fast during manifest preparation with `accelerator selector conflict: key exists with different value` (or equivalent) instead of overwriting.
   - Add tests covering:
     - Pod injection,
     - Deployment or Job controller template injection (at least one),
     - VCJob injection,
     - prepare-time conflict rejection for duplicate selector key with incompatible values using the explicit failure message,
     - no-op when no accelerator profile is present.

## 18. Risks and Tradeoffs

- Profile schema can become an uncontrolled policy surface; keep keys minimal and audited.
- Node selectors/labels are only as correct as scheduler-domain conventions and node labeling hygiene.
- This slice does not solve live hardware validation (`GPU-016`) or usage telemetry certainty (`GPU-018`).
- Tight reuse of existing dispatch order reduces risk but requires careful order checks to avoid override by later injectors.
