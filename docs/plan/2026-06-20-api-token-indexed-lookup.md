# API Token Indexed Lookup

## 1. Objective

Stop API token authentication from scanning every stored token. New user API
tokens must carry their token record id, and verification must perform one store
lookup plus one hash check.

## 2. Background

`problem.md` lists API token verification as a P0 GA blocker because token auth
currently iterates token records and verifies hashes until one matches.

## 4. Assumptions

- Beta tokens without an embedded token id may be rotated before GA.
- Token record ids remain generated by the existing `RecordStore.NextID`.
- The generic store `Get(resource, id)` is the lookup boundary for this slice.

## 5. Non-Goals

- No typed API-token table migration.
- No new token service, cache, or custom index abstraction.
- No JWT/OIDC replacement.
- No legacy scan fallback.

## 6. Current Behavior

Local platform auth and identity auth repository verification call
`List(api_tokens)` and compare token hashes in a loop.

## 7. Target Behavior

User API tokens are formatted as `nexuspaas_<token-id>_<secret>`. Verification
parses `<token-id>`, loads only that record, rejects revoked/expired records,
verifies the stored hash against the full raw token once, loads the active user,
and updates `last_used_at` on success.

## 8. Affected Domains

- Platform request authentication.
- Identity API token creation and verification.
- API token tests.

## 10. API / Contract Changes

API token create responses return the new raw token format. Existing metadata
shape stays the same. Tokens without the indexed format fail closed.

## 11. Database / Migration Changes

None in this slice. The existing token record id is used as the lookup key.

## 12. Configuration Changes

None.

## 13. Observability Changes

No new metrics. Existing request logs and auth failures remain unchanged.

## 14. Security Considerations

The full raw token is still stored only as a hash. The token id alone is not a
credential. Malformed, expired, revoked, missing-user, and wrong-secret tokens
fail closed.

## 15. Implementation Steps

1. Add small platform helpers to format and parse user API tokens.
2. Change identity token creation to generate a secret after the token id is
   allocated and hash the full formatted token.
3. Change identity repository verification to parse id and call `Get`.
4. Change platform local API token auth to parse id and call `Get`.
5. Add tests proving valid tokens work, malformed/legacy tokens fail closed, and
   verification does not call `List`.

## 17. Rollback Plan

Revert this slice. Previously issued new-format tokens would no longer verify
after rollback, so rollback requires users to recreate API tokens.

## 18. Risks and Tradeoffs

Legacy beta tokens fail closed because adding a scan fallback would preserve the
GA blocker. This is acceptable before External GA with a documented token
rotation notice.

## 19. Reviewer Checklist

- Verification path uses one `Get`, not `List`.
- Raw token is not stored.
- Malformed and legacy tokens fail closed.
- Existing token create/list/revoke behavior remains compatible except raw token
  format.
- No new infrastructure or custom cache is introduced.
