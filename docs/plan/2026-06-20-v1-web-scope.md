# V1 Web UI Scope Slice

## 1. Objective

Resolve the `WEB-*` v1 blocker by explicitly scoping the first launch as
API/CLI-first and not advertising a management Web UI in v1.

## 2. Background

`gap-analysis.md` states that a first launch advertising a Web UI needs minimum
Web UI acceptance criteria. This repository does not contain a frontend app or
package manifest. Building a complete OIDC/project/config/job/image/usage UI in
this iteration would be a large new product surface. The practical first launch
should keep API/CLI workflows and WebRTC browser sessions, while deferring the
management Web UI until it has its own slice and acceptance evidence.

## 4. Assumptions

- V1 launch can be API/CLI-first and still preserve GA's later "CLI or Web UI"
  target.
- Browser-operated WebRTC sessions remain in scope as workload streaming, not
  as a full management Web UI.
- Operator/admin integration-proxy UIs such as Harbor/Grafana/MinIO are external
  upstream consoles, not a NexusPaaS end-user management Web UI.

## 5. Non-Goals

- No frontend framework, Web UI scaffolding, or OIDC browser app in this slice.
- No changes to backend API contracts.
- No removal of GA Web UI aspirations.

## 6. Current Behavior

- `gap-analysis.md` lists `WEB-*` under Blocks-v1, but also says those ACs apply
  to a first launch that advertises a Web UI.
- No frontend app is present in the repository.
- Backend integration-proxy routes expose upstream product consoles, not a
  NexusPaaS management UI.

## 7. Target Behavior

- V1 checklist states `WEB-*` is not applicable because v1 does not advertise a
  NexusPaaS management Web UI.
- `gap-analysis.md` records this v1 scope decision while keeping `WEB-*` as
  required when a Web UI is later advertised.
- The next Web UI launch must satisfy `WEB-001..007` before being advertised.

## 14. Security Considerations

surface. Browser WebRTC session security remains covered by the existing RTC
acceptance family.

## 18. Risks and Tradeoffs

- Users expecting a first-party management Web UI must use CLI/API workflows for
  v1.
- This keeps launch scope honest and avoids a rushed UI that cannot satisfy
  authentication, RBAC, ConfigFile, job, image, stream, and usage ACs.
