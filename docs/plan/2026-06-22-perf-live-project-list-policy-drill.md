# PERF Live Project List Policy Drill

## 1. Objective

Unblock `PERF-001` and `PERF-002` evidence for `GET /api/v1/projects` by running the existing k6 100 VU core-read harness with 100 temporary, least-privilege static API key principals that can pass only the Project list route.

## 2. Background

The reusable k6 harness already supports `NEXUSPAAS_PERF_API_KEYS` and deterministic per-VU key selection. The previous live single-key 100 VU run failed with HTTP `429`, proving the limiter keys authenticated traffic by verified principal. The follow-up 100-principal Secret drill avoided rate-limit concentration but failed during setup because each temporary principal received a non-route scope (`platform:perf:read`) and no matching raw PDP policy for `GET /api/v1/projects`.

The first approved execution attempt also proved that the current static admin key can pass gateway admin bootstrap policy but cannot mutate raw policies through `POST /api/v1/permissions/policy`: the authorization-policy handler's `requireAdmin` checks the identity projection/read model for an admin-panel role, not the static principal's `admin=true` header. Therefore this revision uses a tightly scoped one-off DB admin process to insert and delete only the exact temporary raw policy rows.

Code inspection showed the current source would generate:

- resource: `org-project-service:projects`
- operation id: `get_org_project_service_api_v1_projects`
- policy decision tuple: `[subject, project_id query value, route resource, route operation id]`
- scope gate candidates include the route operation id before PDP enforcement

Live diagnostics then found the currently deployed image has a different historical operation id in the admin bootstrap raw policy:

- live resource: `org-project-service:projects`
- live operation id/action: `get_org-project-service_api_v1_projects`

The drill must therefore derive the Project list policy action from the live admin bootstrap row before creating temporary principals. This avoids hardcoding either the current-source value or the currently deployed historical value.

## 4. Assumptions

- The current Kubernetes context remains the reviewed local live context and namespace `nexuspaas`.
- `platform-gateway-runtime-secret` and `org-project-service-runtime-secret` remain the runtime Secrets for gateway and Project service static API keys and principals.
- The existing original static admin key remains valid and must be preserved without printing its value.
- k6 `v1.6.1` is available locally.
- `jq`, `kubectl`, `curl`, `openssl`, `base64`, and `make` are available.
- `psql` is available inside `deployment/postgres`, using `POSTGRES_USER`, `POSTGRES_DB`, and `POSTGRES_PASSWORD` from the container environment.
- The drill may temporarily add 100 static API keys, 100 matching principal bindings, and 100 raw permission policy rows, then must remove the exact temporary policy rows and restore the original Secret bytes for every touched runtime Secret.

## 5. Non-Goals

- Do not make any temporary perf principal an admin.
- Do not use wildcard scopes, `admin`, or broad `projects:read` when the exact operation id scope is sufficient.
- Do not change gateway middleware, PDP behavior, rate limiter behavior, or service code.
- Do not commit generated keys, decoded Secrets, policy row dumps, or k6 summary artifacts.
- Do not use DB writes for any row outside the exact temporary `authorization-policy-service:permission_policies` IDs generated by this drill.
- Do not claim full `PERF-001..008`; this slice only targets the Project list 100-principal read path.

## 6. Current Behavior

- Single-key `100` VU Project list load triggers HTTP `429`.
- Multi-key harness parsing and per-VU key selection pass smoke tests.
- A 100-principal live Secret drill restores the original Secret byte-for-byte but `/api/v1/projects` preflight returns `403`.
- A live-derived tuple drill against only `platform-gateway-runtime-secret` changes the Project preflight from `403` to `401`, proving gateway policy passes but downstream `org-project-service` does not recognize the temporary API key.
- `problem.md` and `gap.md` record that Project list performance evidence is blocked by least-privilege authorization policy.

## 7. Target Behavior

- Each temporary perf principal has only the exact live-derived Project list operation-id scope.
- Each temporary perf principal has exactly one matching raw PDP allow policy:
  `[principal_id, "", "org-project-service:projects", live_project_list_operation_id]`.
- A positive preflight with a temporary key returns HTTP `200` for `GET /api/v1/projects`.
- A negative preflight with the same temporary key returns HTTP `403` for an unrelated protected read route such as `GET /api/v1/groups`.
- `make -C backend perf-k6-core-read` with `100` VUs and 100 keys runs against live gateway with no `429` and no `403` on `/api/v1/projects`.
- All temporary policy rows are deleted and the original Secret bytes are restored after the run, regardless of pass or fail.

## 8. Affected Domains

- Platform gateway runtime authentication and scope gate, by temporary runtime config only.
- Authorization policy raw PDP store, by temporary policy rows only.
- Org Project read API, as the measured endpoint.
- Performance evidence and GA checklist tracking.

No new service boundary is introduced.

## 11. Database / Migration Changes

No schema change. The drill creates and deletes exact temporary raw policy rows in `platform_records`:

- resource: `authorization-policy-service:permission_policies`
- id: `subject || '\x1f' || '' || '\x1f' || 'org-project-service:projects' || '\x1f' || live_project_list_operation_id`
- payload keys mirror `rawPolicyRecord`: `id`, `policy`, `v0`, `v1`, `v2`, `v3`, `sub`, `dom`, `obj`, `act`

The DB admin process must:

- keep the complete generated policy ID list in memory for the whole drill;
- use transaction-wrapped SQL with `ON_ERROR_STOP=1`;
- insert from the complete generated row list only;
- delete by joining against the complete generated ID list only;
- never delete by prefix alone;
- verify post-insert exact-row count equals `100`;
- verify post-delete exact-row count equals `0`.

## 14. Security Considerations

- Do not print, persist, or commit API keys or decoded Secret values.
- Use exact operation id scope rather than wildcard, admin, or broad resource scope.
- Use exact raw policy rows per temporary subject instead of bypassing PDP.
- Keep the dual authorization gate: principal scope must pass and PDP raw policy must match.
- Delete exact policy rows before restoring the Secret.
- Run DB mutation as an admin process with transaction-wrapped SQL and an exact generated ID list; do not print DB passwords, Secret values, or API keys.
- Treat direct DB mutation as a temporary authorization-data repair for this drill only; it intentionally avoids product code changes but must remain tightly scoped and fully reversible.
- Restore the original Secret bytes even if k6 or preflight fails.
- Verify an unrelated protected route remains denied for a temporary perf principal.

## 15. Implementation Steps

1. Capture current live guardrails:
   - current Kubernetes context
   - namespace `nexuspaas`
   - Secret UIDs for gateway and org-project runtime Secrets
   - gateway and org-project readiness
   - count of original API keys and principals in both runtime Secrets
2. Derive the live Project list policy action by querying exactly one existing admin bootstrap policy row where:
   - resource is `authorization-policy-service:permission_policies`
   - payload `sub` is `admin`
   - payload `obj` is `org-project-service:projects`
   - payload `act` matches a Project list action, currently expected to be `get_org-project-service_api_v1_projects` in live
3. Decode original `API_KEYS` and `API_KEY_PRINCIPALS` only into in-process shell variables with tracing disabled.
4. Generate 100 temporary random API keys and deterministic principal IDs prefixed with a unique drill stamp.
5. Build updated comma-separated `API_KEYS` and JSON `API_KEY_PRINCIPALS` in memory.
6. Patch only the two auth Secret fields on both runtime Secrets and restart `deployment/platform-gateway` and `deployment/org-project-service`.
7. Build an in-memory JSON row list with each generated policy `id` and payload, plus a separate in-memory JSON array of exact policy IDs.
8. Use `kubectl exec -i deployment/postgres -- psql -v ON_ERROR_STOP=1` as a one-off admin process to run a transaction that inserts one raw policy row per temporary principal into `platform_records`.
   - The SQL must read only the provided row list.
   - `ON CONFLICT (resource, id) DO NOTHING` is acceptable for re-entrant safety.
   - The transaction must report exact matched-row count after insert.
9. Run positive and negative preflights with one temporary key.
10. Run:
   `NEXUSPAAS_PERF_API_KEYS=<100-temp-keys> NEXUSPAAS_PERF_VUS=100 NEXUSPAAS_PERF_DURATION=30s make -C backend perf-k6-core-read`
11. Delete every exact temporary raw policy row through transaction-wrapped SQL that joins `platform_records` against the exact generated ID array and the resource guard.
12. Verify the post-delete exact ID count is `0`.
13. Restore original Secret base64 values for both runtime Secrets and restart gateway plus org-project deployments.
14. Verify original Secret byte equality, original counts, both deployments' readiness, and no temporary k6 summary artifacts remain.
15. Update `gap.md`, `problem.md`, and this plan with evidence and reviewer outcome.

## 17. Rollback Plan

If any step fails:

1. Stop k6 if still running.
2. Re-run the exact-ID DB delete transaction using the preserved generated policy ID list and resource guard. This cleanup is idempotent: deleting an already-deleted row is acceptable, but post-delete exact ID count must be `0`.
3. Patch the original base64 Secret fields back exactly for both runtime Secrets.
4. Restart `deployment/platform-gateway` and `deployment/org-project-service`, then wait for rollout readiness.
5. Re-check Secret byte equality and original counts for both runtime Secrets.
6. If the delete transaction fails, keep retrying the exact-ID cleanup after restoring gateway Secret state; if it still fails, record the exact count of remaining temporary rows and leave the goal blocked for manual DB intervention.
7. Record the blocker in `problem.md` and leave `PERF-001`/`PERF-002` open.

## 18. Risks and Tradeoffs

- A route or operation id mismatch would still produce `403`; the plan records exact source-derived values to reduce this risk.
- Creating 100 policy rows directly in `platform_records` bypasses authorization-policy API events, but the rows are exact temporary PDP inputs and the supported admin API is blocked by identity-projection admin requirements in the current live environment. The mitigation is transaction-wrapped SQL, complete exact ID lists, post-insert/post-delete counts, and no prefix-only deletes.
- The gateway restart creates a short local-live disturbance; the Secret is restored byte-for-byte afterward.
- Passing this drill proves Project list 100-principal read load, not queue stress, usage-query load, WebRTC concurrency, build concurrency, or full external GA performance.
